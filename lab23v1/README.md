Звіт для Лабораторної роботи №23
1. Аналіз коду до рефакторингу
Порушення DIP: Клас SystemMonitor залежав від конкретних реалізацій (Logger, EmailSender). Будь-яка зміна в цих низькорівневих модулях вимагала перекомпіляції монітора. Клас сам керував життєвим циклом об'єктів (через new), що робило неможливим підміну об'єктів на "фейкові" (Mock) під час тестів.

Порушення ISP: Хоча в даному прикладі монітор використовував усі методи, в реальних системах такий підхід призводить до створення одного гігантського інтерфейсу ISystemServices, де клієнту, якому потрібен лише логер, доводиться реалізовувати методи для відправки пошти та генерації звітів.

2. Опис змін
Створено "вузькі" інтерфейси: ILogger, INotifier, IReportGenerator. Це дозволило розділити обов'язки (ISP).

Впроваджено Dependency Injection: Клас SystemMonitor тепер приймає інтерфейси через конструктор. Це реалізація принципу DIP — модулі вищого рівня не залежать від деталей, вони залежать від абстракцій.

3. Висновки
Застосування ISP та DIP значно підвищило гнучкість системи. Тепер ми можемо легко замінити EmailNotifier на TelegramNotifier, просто створивши новий клас, що реалізує інтерфейс, не змінюючи при цьому логіку SystemMonitor. Код став чистішим, легшим для тестування та підтримки.
