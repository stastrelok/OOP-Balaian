Звіт з аналізу SOLID принципів (SRP, OCP) в AutoMapper
1. Обраний проєкт
Назва: AutoMapper

Посилання на GitHub: https://github.com/AutoMapper/AutoMapper

2. Аналіз SRP (Single Responsibility Principle)
2.1. Приклади дотримання SRP
Клас: MapperConfiguration
Відповідальність: Валідація та створення плану мапінгу на основі профілів.

Обґрунтування: Цей клас не займається самим перетворенням об'єктів. Його єдина причина для зміни — зміна алгоритму того, як конфігурація збирається докупи або як перевіряється її цілісність.


public class MapperConfiguration : IConfigurationProvider {
    // Логіка збору профілів та валідації конфігурації
    public void AssertConfigurationIsValid() { ... }
}
Клас: ResolutionContext
Відповідальність: Передача метаданих та стану під час процесу мапінгу.

Обґрунтування: Це типовий "об'єкт-контейнер". Він не знає, як мапити дані, він лише зберігає контекст, необхідний для роботи інших компонентів.


public class ResolutionContext {
    public object SourceValue { get; }
    public object DestinationValue { get; }
    // Тільки властивості та методи доступу до даних контексту
}
2.2. Приклад порушення SRP
Клас: TypeDetails (внутрішній)
Множинні відповідальності: Аналіз типу через рефлексію, пошук конструкторів, фільтрація публічних/приватних властивостей та кешування цих результатів.

Проблеми: Цей клас є "Reflection God" для конкретного типу. Якщо зміниться логіка того, як ми хочемо бачити конструктори, або логіка кешування — доведеться змінювати один і той самий файл. Це ускладнює тестування аналізу типів окремо від їхнього кешування.

3. Аналіз OCP (Open/Closed Principle)
3.1. Приклади дотримання OCP
Сценарій: Користувацькі конвертори (ITypeConverter<TSource, TDestination>)
Механізм розширення: Інтерфейс.

Обґрунтування: AutoMapper "закритий" для модифікації свого ядра кожного разу, коли комусь потрібно перетворити String на DateTime особливим чином. Ви просто реалізуєте інтерфейс у своєму коді, і бібліотека підхоплює його.


public interface ITypeConverter<in TSource, TDestination> {
    TDestination Convert(TSource source, TDestination destination, ResolutionContext context);
}
Сценарій: Профілі конфігурації (Profile)
Механізм розширення: Абстрактний клас.

Обґрунтування: Ви розширюєте функціональність мапінгу у своєму проєкті, створюючи нові класи, що успадковуються від Profile, не змінюючи код самої бібліотеки AutoMapper.

3.2. Приклади порушення OCP
Сценарій: Внутрішні методи перевірки примітивних типів
Проблема: У глибинах ядра (наприклад, у логіці PrimitiveHelper) іноді зустрічаються жорстко прописані перевірки на базові типи .NET.

Наслідки: Якщо в майбутніх версіях .NET з'являться нові примітивні типи (як це було з Half або новими числовими типами), розробникам AutoMapper доводиться вручну додавати нові if або switch блоки в ядро, замість того, щоб мати плагінну систему для визначення "примітивності" типу.

4. Загальні висновки
AutoMapper — це еталонний приклад застосування SOLID у великих масштабах. Використання інтерфейсів для кожного кроку (від валідації до розв'язання значень) робить систему надзвичайно гнучкою. Більшість порушень принципів у репозиторії пов'язані з оптимізацією продуктивності (рефлексія — це повільно, тому доводиться створювати складніші, менш "чисті" класи для кешування) або підтримкою застарілих версій.

В цілому, архітектура проєкту дозволяє розширювати його на 99% без втручання в основний код, що і є головною перемогою OCP.
