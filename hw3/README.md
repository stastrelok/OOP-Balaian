Принципи ISP та DIP: Шлях до гнучкої архітектури
У розробці складного програмного забезпечення ми часто стикаємося з проблемою "жорсткості" коду, коли зміна в одному модулі ламає інший. Щоб уникнути цього, використовуються принципи SOLID, серед яких особливе місце посідають ISP (розділення інтерфейсів) та DIP (інверсія залежностей).

1. Принцип розділення інтерфейсів (ISP)
ISP стверджує: клієнти не повинні залежати від методів, які вони не використовують. Краще мати багато специфічних ("вузьких") інтерфейсів, ніж один універсальний ("товстий").

Приклад порушення ISP
Розглянемо інтерфейс для багатофункціонального пристрою:


public interface IMachine
{
    void Print(Document d);
    void Scan(Document d);
    void Fax(Document d);
}
Якщо ми захочемо реалізувати клас OldPrinter, який вміє лише друкувати, нам доведеться залишити методи Scan та Fax порожніми або викидати NotImplementedException. Це змушує клас залежати від функціоналу, якого він не має.

Вирішення проблеми
Розділимо "товстий" інтерфейс на три вузьких:


public interface IPrinter { void Print(Document d); }
public interface IScanner { void Scan(Document d); }
public interface IFax { void Fax(Document d); }

// Тепер кожен клас бере лише те, що йому потрібно:
public class OldPrinter : IPrinter {
    public void Print(Document d) { /* логіка друку */ }
}

public class MultiFunctionDevice : IPrinter, IScanner, IFax {
    /* реалізація всіх методів */
}
2. Принцип інверсії залежностей (DIP)
DIP каже: модулі верхнього рівня не повинні залежати від модулів нижнього рівня. Обидва мають залежати від абстракцій.

Переваги застосування DIP через Dependency Injection (DI)
Замість того, щоб створювати об'єкт всередині класу (new DataService()), ми передаємо його ззовні (через конструктор).

Переваги:

Гнучкість (Loosely Coupling): Ми можемо замінити реалізацію БД (наприклад, з SQL на MongoDB), не змінюючи жодного рядка в бізнес-логіці.

Масштабованість: Легко додавати нові функції, просто створюючи нові класи-реалізації інтерфейсів.

Керування життєвим циклом: DI-контейнери самі вирішують, коли створювати об'єкт і коли його видаляти.

3. Як "вузькі" інтерфейси (ISP) сприяють кращому DI та тестуванню
ISP та DIP працюють у тандемі. Коли ми використовуємо "вузькі" інтерфейси, ми отримуємо значні переваги в архітектурі:

Точність впровадження (DI): Клас отримує рівно ту залежність, яка йому потрібна. Якщо методу потрібен лише друк, ми передаємо IPrinter, а не весь IMachine. Це робить наміри коду прозорими.

Спрощення тестування (Unit Testing): Для тестування класу нам потрібно створити "заглушку" (Mock). Створити мок-об'єкт для інтерфейсу з одним методом значно легше, ніж для інтерфейсу з десятьма. Ми уникаємо написання зайвого коду для методів, які взагалі не стосуються тесту.

Зменшення перекомпіляції: Якщо ми змінили логіку в IScanner, класи, що залежать тільки від IPrinter, не потребуватимуть перекомпіляції чи тестування, оскільки вони ізольовані.

Висновок
Дотримання ISP робить інтерфейси зрозумілими та мінімалістичними, а застосування DIP звільняє модулі від жорстких зв'язків. Разом ці принципи перетворюють монолітний код на "конструктор LEGO", де кожну деталь можна легко замінити або протестувати окремо.
